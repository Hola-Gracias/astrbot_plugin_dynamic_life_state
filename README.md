# DynamicLifeState - 动态生活状态插件

AstrBot 插件，用于按可配置的业务周期为 Bot 生成生活状态时间线，并在每次 LLM 请求前注入当前时段的日程与穿搭。

## 功能特性

- **周期状态生成**：按 `generate_time` 划分业务周期，每个周期生成一份完整生活状态
- **自动生成与补偿**：支持每日定时生成；缺少当前状态时，会在首次对话或查询时自动补充生成
- **动态时段匹配**：支持自然时段和具体时间区间，并根据当前时间选择对应日程与穿搭
- **历史状态参考**：生成新状态前归档旧状态，可参考最近若干个成功业务周期
- **临时上下文注入**：默认通过临时用户内容注入，不写入 AstrBot 对话历史
- **完整状态查询**：提供指令和 LLM Tool，可查询当前或指定业务日期的完整状态
- **会话范围控制**：支持白名单、黑名单和全部启用三种模式
- **失败保护**：状态归档、生成和持久化分别处理，存储失败时保留已有状态

## 安装

通过 AstrBot 插件管理界面安装，或将本插件目录放置到 AstrBot 的插件目录中。

插件依赖 `APScheduler`，通过 AstrBot 安装插件依赖时会根据 `requirements.txt` 自动安装。

## 配置项

安装后可在 AstrBot WebUI 中配置。`generate_time`、时区和定时任务会在插件初始化时解析。

| 配置项 | 类型 | 默认值 | 说明 |
| -------- | ------ | -------- | ------ |
| `session_list_mode` | string | `whitelist` | 会话范围模式：`none` 全部启用，`whitelist` 仅列表启用，`blacklist` 列表禁用 |
| `session_list` | list | `[]` | 会话 UMO 列表，例如 `default:GroupMessage:123456`，可通过 `/sid` 获取 |
| `generate_time` | string | `07:00` | 每日生成时间，同时也是业务周期边界，格式为 `HH:MM` |
| `history_reference_days` | int | `3` | 生成时参考的最近成功历史业务周期数，范围为 0–7；设为 0 时不参考历史状态 |
| `llm_provider_id` | string | `""` | 用于生成状态的 LLM Provider；留空时使用当前默认 Provider |
| `persona_id` | string | `""` | 用于生成状态的人格设定；留空时使用当前默认人格 |
| `injection_mode` | string | `extra_user_content_parts` | 状态注入方式：`extra_user_content_parts` 或 `fake_tool_call` |
| `prompt_template` | text | 内置模板 | 状态生成提示词；留空时使用内置模板 |
| `debug_mode` | bool | `false` | 输出生成提示词、模型原始返回和注入内容等调试日志 |

默认使用白名单模式。如果 `session_list` 为空，插件不会向任何会话注入状态，LLM Tool 也会返回当前会话未启用。

### 提示词占位符

| 占位符 | 内容 |
| -------- | ------ |
| `{business_date}` | 当前业务日期，格式为 `YYYY-MM-DD` |
| `{cycle_start}` | 业务周期开始时间，包含时区偏移 |
| `{cycle_end}` | 业务周期结束时间，包含时区偏移 |
| `{persona}` | 选定人格的系统提示词 |
| `{history_states}` | 最近成功历史状态的渲染文本；无可用历史时为“无” |
| `{extra_requirements}` | `/dls regen` 附带的额外要求；无额外要求时为“无” |

内置模板将历史状态和额外要求放在提示词末尾，以尽量保持前缀稳定。自定义模板不会自动补齐缺失的占位符。

## 指令

当前已实现指令的权限如下：

| 指令 | 权限 | 说明 |
| ------ | ------ | ------ |
| `/dls show` | 管理员 | 查看当前时间对应的时段状态 |
| `/dls show <时段>` | 管理员 | 查看指定自然时段，例如“上午”“晚上” |
| `/dls show <HH:MM>` | 管理员 | 查看当前业务周期内指定时间点的状态 |
| `/dls full` | 管理员 | 查看当前业务日期的概况、氛围和完整时间线 |
| `/dls full <YYYY-MM-DD>` | 管理员 | 查看指定业务日期的完整状态；仅查询已有数据，不生成历史状态 |
| `/dls regen [额外要求]` | 管理员 | 强制重新生成当前业务日期的状态，可附加整段自然语言要求 |

## LLM Tool

插件注册 `get_full_dynamic_life_state` Tool。仅当用户明确询问全天安排、完整日程、其他时段状态或历史状态时，模型才应调用该工具。

- 不传 `date`：返回当前业务周期的完整状态
- 传入 `YYYY-MM-DD`：查询对应业务日期的当前或历史状态
- Tool 与自动注入共用会话范围配置

## 工作原理

### 状态生成流程

1. 根据 AstrBot 全局时区、当前时间和 `generate_time` 计算业务日期与周期边界
2. 生成前将其他业务日期的当前状态归档到 `history/`
3. 读取早于目标日期的最近 N 个成功历史状态
4. 将业务周期、人格、历史状态和额外要求渲染到提示词
5. 调用配置的 LLM Provider，并校验返回的 JSON 结构、业务日期和时间段
6. 原子写入当前状态；同一业务日期重新生成时直接替换旧状态

状态可能由以下入口触发生成：

- 每日 `generate_time` 定时任务
- 当前业务周期没有有效状态时的首次 LLM 请求
- `/dls show` 或 `/dls full` 查询当前状态时
- `/dls regen` 强制重新生成

### 状态注入流程

每次 LLM 请求前，插件会先检查会话范围，再选择当前业务周期和当前时段：

- `extra_user_content_parts`：以临时 `TextPart` 注入 `<life_state>`，不写入 conversation history，推荐使用
- `fake_tool_call`：向请求上下文追加伪造工具调用消息对；Gemini Provider 会自动降级为 `extra_user_content_parts`

注入内容只包含当前时段所需的信息。全天状态和其他时段安排由 `get_full_dynamic_life_state` 按需查询。

### 数据存储

```text
data/plugin_data/astrbot_plugin_dynamic_life_state/
├── life_state.json
└── history/
    ├── life_state_2026.06.29.json
    └── life_state_2026.06.30.json
```

- `life_state.json` 只保存当前业务日期的状态
- 历史文件名中的日期为业务日期，每个文件只保存一个业务日期
- 最多保留最近 7 个有效历史文件；损坏或日期不匹配的历史文件会在清理时删除
- 生成提示词只参考 `status == "ok"` 且早于目标日期的历史状态
- `/dls full <YYYY-MM-DD>` 和 LLM Tool 均可查询保留期内的历史状态

### 状态数据格式

LLM 必须返回可解析的 JSON 对象，至少包含以下字段：

```json
{
  "business_date": "2026-06-30",
  "schedule_summary": "该业务周期的整体日程",
  "style_summary": "该业务周期的整体穿搭或氛围",
  "timeline": [
    {
      "time": "上午",
      "schedule": "处理资料和消息",
      "outfit": "浅色针织衫、牛仔裤"
    },
    {
      "time": "14:00-17:00",
      "schedule": "去图书馆查资料",
      "outfit": "浅色针织衫、牛仔裤"
    }
  ]
}
```

`timeline` 至少需要一个可解析条目；内置提示词会要求模型生成至少 3 个时段。每个有效条目必须包含 `time`、`schedule` 和 `outfit`。

## 业务周期与时段规则

业务日期不是自然日。例如 `generate_time=07:00` 时，业务日期 `2026-06-30` 对应：

```text
[2026-06-30 07:00, 2026-07-01 07:00)
```

因此该业务日期中的“凌晨”位于次日自然日期，即 `2026-07-01 00:00–06:00`。状态生成后会冻结当时的周期边界，后续修改 `generate_time` 不会重新解释已有状态。

### 自然时段映射

| 时段 | 区间 |
| ------ | ------ |
| 凌晨 | 00:00–06:00 |
| 早上 | 06:00–09:00 |
| 上午 | 09:00–12:00 |
| 中午 | 12:00–14:00 |
| 下午 | 14:00–18:00 |
| 傍晚 | 18:00–20:00 |
| 晚上 | 20:00–23:00 |
| 深夜 | 23:00–24:00 |

所有区间均为左闭右开。例如 12:00 属于“中午”，14:00 属于“下午”。

### 匹配优先级

1. **具体时间区间**：`12:30-13:30` 等具体区间覆盖查询时间时优先使用
2. **自然时段**：使用该自然时段在业务周期内按时间排序的第一个有效片段
3. **空缺回退**：没有匹配条目时，临时返回 `schedule="空闲"`，穿搭继承周期内最近的更早时段

自然时段被业务周期边界切出的周期尾部残片不会重复使用同一条目。例如周期结束前的 `06:00–07:00` 如需精确安排，应在时间线中增加具体时间区间。

## 注意事项

- 所有会话共享一份全局生活状态，不按会话、群组或人格分别存储
- 修改人格或 Provider 不会重写已有状态，需要使用 `/dls regen` 或等待下一业务周期
- 自定义提示词应保留 JSON 输出约束；模型返回无法解析、日期不匹配或时间线无有效条目时，本次状态会标记为生成失败
- 插件使用 AstrBot 全局时区；时区无效时回退到 `Asia/Shanghai`
- `generate_time` 必须使用严格的 `HH:MM` 格式；无效配置会回退到 `07:00`
- 插件不读取聊天记录，也不集成天气、节日或外部日程 API

## 兼容性

- 在 AstrBot v4.26.4 环境验证
- 状态生成需要至少一个可用的文本 LLM Provider
- 功能与平台适配器无关，但 `fake_tool_call` 的支持情况取决于 Provider
